## Executive Summary

WSOP TV의 프리미엄 시청 경험을 제공하는 **Advanced Mode** 기능 정의. Plus+ ($50) 구독자 전용으로 **Multi-view**와 **StatsView** 두 가지 핵심 기능을 통해 경쟁사(PokerGo) 대비 차별화된 가치를 제공합니다.

### 핵심 목표
- 아이돌 직캠 스타일의 Multi-view로 몰입감 극대화
- HUD 스타일 StatsView로 전문적인 시청 경험 제공
- Plus+ 구독의 가치 제안 확립

---

## Problem Statement

### 현재 상황
- PokerGo 등 경쟁 서비스는 단일 화면 스트리밍만 제공
- 포커 팬들은 특정 플레이어에 집중하고 싶어함
- 통계 정보 없이는 전문적인 분석 시청이 어려움

### 해결 방안
- **Multi-view**: 메인 화면 + 개별 플레이어 직캠 동시 제공
- **StatsView**: 플레이어 통계 HUD 오버레이 제공
- 실시간 View Mode 전환으로 유연한 시청 경험

## Feature Specification

### 1. Multi-view (멀티뷰)

#### 1.1 개념 정의

> "메인화면이 중앙에 있고, 각 유저들의 얼굴을 잡고 있는 화면이 옆에 또 있는 방식 (아이돌 직캠 스타일)"
> — Michael Note

#### 1.2 프로덕션 스타일과의 차이점

**현재 프로덕션 스타일** (PGM - 기존 방송):
- 여러 개의 피처 테이블 및 콘텐츠 소스를 조합하여 최종 방송(PGM)으로 송출
- PD가 의도한 연출에 따라 단일 화면 제공
- 모든 시청자가 동일한 화면 구성 경험

**Multi-view 스타일** (WSOP TV 전용):
- 메인 PGM (기존 방송) 유지
- 추가로 개별 플레이어 직캠 동시 제공
- 사용자가 관심 플레이어 선택 가능

#### 1.3 핵심 쟁점

**Multi-view 도입 시 결정사항**:

| 쟁점 | 설명 | 영향 |
|------|------|------|
| **쟁점 1: 다중 피처 테이블 지원** | 피처 테이블이 변경될 때마다 다른 테이블의 다른 화면들까지 함께 보여줄 것인가? | 멀티캠 인프라 확대, 스트림 수 증가 |
| **쟁점 2: 플레이어 선택 범위** | 특정 피처 테이블의 특정 선수만 보여줄 것인가? 아니면 모든 테이블의 모든 선수를 보여줄 것인가? | UX 복잡도, 대역폭 비용 |

**원래 목적 (아이돌 직캠 스타일의 니즈)**:
- 포커 팬들은 자신이 좋아하는 선수만 집중해서 보고 싶은 경향
- 따라서 **쟁점 2번 (특정 선수만)** 형태가 원래 목적과 일치

**결론 - WSOP TV 제공 방식**:
1. **Standard Mode**: 기존 PGM (모든 사용자)
2. **Multi-view Mode**: WSOP TV 전용 + 피처 테이블 선수 직캠 (Plus+ 전용)

이를 통해 Plus+의 진정한 가치 = "좋아하는 선수만 집중 시청" 제공

#### 1.3.1 개선 아이디어 및 해결방안

**Multi-view 쟁점을 해결하기 위한 구체적인 개선 아이디어**:

##### 아이디어 1: "플레이어 팔로우" 모드
- **설명**: 사용자가 원하는 플레이어를 사전에 "팔로우"하면, 해당 선수가 속한 테이블의 직캠만 표시
- **장점**:
  - 사용자 선택 범위 명확 (쟁점 2 해결)
  - 대역폭 효율적 (팔로우한 선수의 캠만 수신)
  - 개인화된 경험 제공
- **구현 복잡도**: 낮음 (캠 선택 로직)
- **우선순위**: P1

##### 아이디어 2: "테이블 뷰 패키지"
- **설명**: 사용자가 한 번에 선택 가능한 테이블 조합 제공
  - 예: "Final Table (9캠)" vs "Day 2 - Table A (6캠)" vs "Custom Mix"
- **장점**:
  - UX 단순화 (테이블 선택 → 자동으로 모든 캠 제공)
  - 피처 테이블 변경 시 대응 용이 (쟁점 1 해결)
  - 비용 예측 가능 (테이블당 인프라 정량화)
- **구현 복잡도**: 중간 (테이블 메타 관리)
- **우선순위**: P1

##### 아이디어 3: "동적 오토매틱 스위칭"
- **설명**: 시스템이 자동으로 Best Action이 일어나는 캠으로 전환
  - All-in, 큰 팟 형성, 대기 시간 등을 감지하여 관련 선수 캠으로 자동 전환
- **장점**:
  - 사용자 개입 최소화 (손 떨림 없음)
  - TV 앱에서도 활용 가능 (리모컨 필요 없음)
  - 프로페셔널한 시청 경험
- **구현 복잡도**: 높음 (Action Detection AI 필요)
- **우선순위**: P2
- **기술 스택**: Hand History Parser + Computer Vision

##### 아이디어 4: "멀티뷰 프리셋 저장"
- **설명**: 사용자가 자주 사용하는 멀티뷰 구성을 저장/로드
  - 예: "Phil Ivey Focus" (Phil만 풀스크린), "Chip Leader Battle" (상위 3명)
- **장점**:
  - 일회성 설정 최소화
  - 다양한 이용 시나리오 지원
- **구현 복잡도**: 낮음 (사용자 프로필 저장)
- **우선순위**: P1

| 기능 | Plus ($10) | Plus+ ($50) |

> "Exclusive Content (behind-the-scenes) 는 굳이 WSOPTV 용으로 따로 제작하지 말자."
> — Michael Note

| Plus+ 전용 기능 | 가치 |
|----------------|------|
| Multi-view | 좋아하는 플레이어 집중 시청 |
| StatsView | 전문적인 분석 시청 |
| (향후) 4K 화질 | 프리미엄 화질 |

## Technical Requirements

### 스트리밍 아키텍처

#### Multi-view 구현 방식

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 클라이언트 믹싱** | 10개 스트림 동시 수신 | 유연한 레이아웃 | 트래픽 10배, 기기 부하 |
| **B. 서버 믹싱** | 서버에서 합성 후 송출 | 트래픽 1배 | 레이아웃 고정 |
| **C. 하이브리드** | 메인+선택 캠만 수신 | 균형 | 복잡도 증가 |

**권장안**: **B. 서버 믹싱**
- 30분+ 의도적 지연 방송이므로 사전 렌더링 가능
- 트래픽 비용 최적화

| 옵션 | 설명 | 실현 가능성 | 비용 |

| **B. 사전 계산 통계** | 역대 데이터 기반 통계 | 높음 | 중간 |
| **C. 수동 입력** | 해설자/운영자 입력 | 높음 | 낮음 |

**권장안**: **B. 사전 계산 통계** (MVP)

- 대회 시작 전 플레이어 프로필 캐싱

## Open Questions

### 프로덕션팀 (PD/연출) 제안 필요

1. **Multi-view 레이아웃 선호도**
   - 직캠 스타일 (메인 + 하단 9개) vs 4분할 그리드?
   - 제안 배경: 아이돌 직캠 vs 스포츠 멀티뷰 두 가지 방향 존재

2. **TV 앱 예외 처리**
   - Plus+ 구독자가 TV에서 Advanced Mode 미지원 시 사용자 경험 어떻게 할 것인가?
   - 제안 배경: 리모컨 UX 제약으로 인한 구현 불가능성

### GG POKER (데이터 제공) 제안 필요

1. **StatsView 통계 범위 및 정확성**
   - GGPoker 내부 통계만 제공 vs 외부 데이터 연동?
   - GTO Wizard 연동 가능성 및 기술 스택?
   - 제안 배경: 통계 데이터의 신뢰성과 확장성

2. **통계 데이터 갱신 주기**
   - 대회 전 1회 캐싱 vs 핸드별 실시간 갱신?
   - 제안 배경: 레이턴시 vs 정확성 트레이드오프

### 개발팀 (기술 구현) 제안 필요

1. **서버 믹싱 vs 클라이언트 믹싱**
   - B안(서버 믹싱) 권장 근거: 30분+ 지연 방송이므로 사전 렌더링 가능
   - 제안 배경: 트래픽 비용 최적화
