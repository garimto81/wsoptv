## 개요

이 문서는 WSOPTV OTT 플랫폼 구축을 위한 Mega Cloud 업체 미팅에서 사용할 질문지입니다. WAVVE 구축 경험을 바탕으로 WSOPTV 핵심 요구사항 충족 가능성을 평가합니다.

### 업체 개요

| 항목 | 내용 |
|------|------|
| **회사** | Mega Cloud (메가클라우드) |
| **소재지** | 한국 |
| **주요 구축 사례** | WAVVE, Vimeo OTT 일부 |
| **레퍼런스 규모** | WAVVE 4.2M MAU (한국 OTT 2위) |

## 질문 목록 (14개)

### 1. 플레이어/SDK (4개)

#### Q1. WAVVE에서 사용한 플레이어 기술 스택은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WAVVE는 한국 최대 OTT 서비스로 4.2M MAU를 처리합니다. 이 규모를 안정적으로 운영한 플레이어 기술이 무엇인지 파악해야 WSOPTV에 적용 가능한지 판단할 수 있습니다. |
| **기술 배경** | 플레이어는 자체 개발(완전한 커스터마이징 가능, 유지보수 부담 큼) 또는 외부 OVP 기반(빠른 개발, 라이선스 비용 발생)으로 구축할 수 있습니다. |
| **확인 포인트** | 자체 개발 vs OVP 기반 / HLS, DASH 지원 / DRM 통합 방식 |

#### Q2. 멀티뷰(다중 카메라) 동시 재생 구현 경험은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WSOPTV의 핵심 기능인 멀티뷰는 메인 테이블 영상과 개별 플레이어 카메라를 동시에 표시해야 합니다. WAVVE가 스포츠 중계 등에서 멀티뷰를 구현한 경험이 있다면 WSOPTV에 적용 가능합니다. |
| **기술 배경** | 멀티뷰는 여러 스트림을 동시에 디코딩하고 렌더링해야 합니다. 대역폭 최적화(저화질 썸네일 스트림 사용)와 스트림 동기화가 기술적 도전입니다. |
| **확인 포인트** | 동시 스트림 수 / 동기화 방식 / 대역폭 최적화 전략 |

#### Q3. Quick VOD (Live→VOD 실시간 전환) 기술 세부사항은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | Quick VOD는 WAVVE의 차별화 기능으로, 라이브 방송 중에도 이미 방송된 부분을 VOD처럼 되감기/빨리감기할 수 있습니다. |
| **기술 배경** | 일반적으로 라이브 스트림은 실시간만 시청 가능하고, VOD는 방송 종료 후 별도 처리가 필요합니다. Quick VOD는 라이브 스트림을 실시간으로 세그먼트 단위로 저장하여 즉시 VOD로 제공합니다. |
| **확인 포인트** | 전환 지연 시간 / 세그먼트 저장 방식 / 스토리지 아키텍처 |

#### Q4. 자막/다국어 지원 아키텍처는?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WSOPTV는 글로벌 서비스로 20개 언어 자막을 지원해야 합니다 (GG POKER 지원 언어 기준). |
| **기술 배경** | 자막은 사전 제작(VOD에 미리 자막 파일 첨부)과 실시간 생성(라이브 중 음성을 텍스트로 변환)으로 나뉩니다. |
| **확인 포인트** | 자막 포맷 (WebVTT, SRT) / 실시간 자막 지원 여부 / 다국어 동시 제공 방식 |

### 2. OTT/앱 (4개)

#### Q5. WAVVE 앱 개발 시 사용한 기술 스택은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | GGPoker 개발팀이 향후 WSOPTV 앱을 유지보수하려면 기술 스택 호환성이 중요합니다. |
| **기술 배경** | Web: React, Vue, Angular 등. Mobile: 네이티브(Swift/Kotlin), 크로스플랫폼(React Native/Flutter). TV: Tizen(삼성), webOS(LG), Roku SDK 등. |
| **확인 포인트** | 플랫폼별 기술 스택 / 코드 공유율 / 유지보수 체계 |

#### Q6. TV 앱 개발 경험 범위는?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WSOPTV는 글로벌 서비스이므로 북미 시장에서 인기 있는 Roku, Fire TV도 필수입니다. |
| **기술 배경** | Samsung Tizen, LG webOS는 웹 기술 기반으로 개발 방식이 유사합니다. Roku는 BrightScript라는 독자 언어를, Fire TV는 Android 기반입니다. |
| **확인 포인트** | 지원 플랫폼 목록 / Roku/Fire TV 개발 경험 유무 / 인증 경험 |

#### Q7. 글로벌 서비스 확장 경험은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WAVVE는 2022년 KOCOWA 인수 및 wavve go 해외 진출 경험이 있습니다. |
| **기술 배경** | 글로벌 서비스는 다국어 앱 관리, 글로벌 인프라 구성, 현지화가 필요합니다. |
| **확인 포인트** | 해외 서비스 지역 / 현지화 수준 / 규제 대응 경험 |

#### Q8. 앱 성능 최적화 경험 (4.2M MAU 처리)은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WAVVE의 4.2M MAU는 한국 OTT 2위 규모입니다. WSOPTV의 50만 동시접속 목표 달성에 중요한 레퍼런스입니다. |
| **기술 배경** | 대규모 트래픽 처리에는 캐싱 전략, 로드밸런싱, 오토스케일링, 데이터베이스 최적화 등이 필요합니다. |
| **확인 포인트** | 피크 트래픽 수치 / 최적화 전략 / 장애 대응 사례 |

---

### 3. 인프라 (3개)

#### Q9. CDN 파트너 및 글로벌 커버리지는?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WAVVE는 한국 시장 중심이므로 한국 CDN 경험이 풍부하나, WSOPTV는 글로벌 CDN이 필수입니다. |
| **기술 배경** | 한국 로컬 CDN은 한국 내에서 빠르지만 해외 PoP가 부족합니다. 글로벌 CDN(Akamai, CloudFront)은 전 세계 PoP를 보유합니다. |
| **확인 포인트** | 현재 CDN 파트너 / 글로벌 PoP 현황 / Multi-CDN 경험 |

#### Q10. 대규모 동시접속 처리 아키텍처는?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WSOPTV는 WSOP Main Event 기간에 50만 동시접속을 목표로 합니다. |
| **기술 배경** | 대규모 동시접속 처리에는 마이크로서비스 아키텍처, 오토스케일링, 로드밸런싱, 캐싱(Redis 등), 메시지 큐(Kafka 등)가 필요합니다. |
| **확인 포인트** | 아키텍처 다이어그램 / 스케일링 전략 / 병목 구간 및 해결책 |

#### Q11. 해외 지역 서비스 시 인프라 구성 방안은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WSOPTV는 글로벌 서비스이므로 인프라도 글로벌하게 구성해야 합니다. |
| **기술 배경** | 멀티 리전 배포는 여러 지역(예: 서울, 버지니아, 프랑크푸르트)에 서버를 배치하여 지연 시간을 최소화합니다. |
| **확인 포인트** | 멀티 리전 경험 / 데이터 동기화 전략 / 장애 복구(Failover) |

---

### 4. 연동/운영 (3개)

#### Q12. 외부 시스템(빌링/CMS/SSO) 연동 API 설계 경험은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | WSOPTV는 GGPoker의 빌링 시스템, 스케줄 데이터, GGPass SSO와 연동해야 합니다. |
| **기술 배경** | WAVVE는 SKT 결제, 통신사 번들 등 다양한 외부 시스템과 연동 경험이 있습니다. |
| **확인 포인트** | API 설계 표준 / 인증 방식 / 에러 처리 및 재시도 로직 |

#### Q13. WAVVE 운영 중 주요 장애 사례와 대응 방식은?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | 어떤 서비스든 장애는 발생합니다. 중요한 것은 얼마나 빠르게 감지하고 복구하는지입니다. |
| **기술 배경** | 장애 대응 역량은 모니터링 시스템, 알람 체계, 장애 대응 프로세스(런북), 사후 분석(포스트모템)으로 평가합니다. |
| **확인 포인트** | 대표 장애 사례 / MTTR(복구 시간) / 재발 방지 조치 |

#### Q14. 글로벌 프로젝트 수행 시 커뮤니케이션 체계는?

| 항목 | 내용 |
|------|------|
| **왜 중요한가** | GGPoker 본사는 영국에 있고, WSOPTV 관련 의사결정자는 영어로 커뮤니케이션합니다. |
| **기술 배경** | 글로벌 프로젝트는 영어 문서화, 화상 미팅(시차 고려), 실시간 채팅(Slack 등), 이슈 트래킹(Jira 등)이 필요합니다. |
| **확인 포인트** | 영어 커뮤니케이션 가능 인원 / 시차 대응 체계 / 문서화 수준 |
